# Contribution Guide

We appreciate your interest in contributing to **VibeHubAI Rules**.
This repository exists to help developers **build better software with AI assistance**, not to collect generic prompts or low-effort rules.

Please read this guide carefully before contributing.

---

## Philosophy

**Quality over quantity ‚Äî always.**

VibeHubAI Rules are treated as **engineering assets**, not prompt dumps.

Our core principles:

* **Real-world tested**
  Rules should be based on actual production experience, not theory.

* **Opinionated and actionable**
  Clear guidance beats vague suggestions.

* **Tool-aware**
  Rules should be written with a specific AI tool in mind (Cursor, Copilot, Windsurf, Kiro, Replit, etc.).

* **Consistency > novelty**
  We prefer proven patterns over trendy experiments.

* **No instant publishing**
  Every contribution is reviewed before acceptance.

## Contribution Process

1. **Fork the repository**
2. Create a new branch:

   ```bash
   feature/add-<tool>-<topic>-rules
   ```
3. Add your rule file in the appropriate directory
4. Ensure naming, format, and metadata follow repository conventions
5. Open a Pull Request with:

   * What problem this rule solves
   * Which tool(s) it targets
   * Any assumptions or constraints

---

## Review Process

* All submissions are manually reviewed
* Feedback may be requested before acceptance
* Maintainers may:

  * Request changes
  * Suggest merging with an existing rule
  * Decline the contribution with rationale

Acceptance is based on **clarity, usefulness, and alignment**, not contributor seniority.
